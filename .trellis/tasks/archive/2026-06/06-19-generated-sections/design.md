# Design — D3b deterministic generated governance page

## 1. Components

```
cli.py  wiki --write ──▶ app/wiki.py::write_generated_sections(repo) -> list[Path]
                              │ inv = run_inventory(repo, exclude_under=(GENERATED,))
                              │ h   = inventory_hash(render_inventory_json(inv))
                              │ md  = render_generated_overview(inv, h)   (pure)
                              │ write .codas/wiki/generated/governance.md
                              ▼
                          committed page (governance map + atlas:claims fence)
```

`render_generated_overview` reuses the same generated-excluded inventory + hash as the
D3a pack (`build_atlas_pack` already does this) — the page and the pack pin the same
`source_inventory_hash`. No adapter import (§11): app/wiki uses app/inventory +
core/provenance only.

## 2. The page (`.codas/wiki/generated/governance.md`)

```markdown
<!-- GENERATED by `codas wiki --write`. Do not edit by hand; regenerate to refresh. -->

# Atlas Governance Overview (generated)

VERIFIED GOVERNANCE FACTS (prefer over inferred structure). Rendered deterministically
from repository facts by `codas wiki --write`; edit the sources of truth
(`.codas/structure.yml`, `.codas/program.yml`) and regenerate.

## Structure Units

| unit | path | kind | owner |
| --- | --- | --- | --- |
| `atlas-wiki` | `.codas/wiki` | orientation_layer | Orientation Curator |
| ... sorted by unit id ... |

## Roadmap

| work item | phase | status |
| --- | --- | --- |
| `program:P0:cli-core` | P0 | completed |
| ... sorted by work-item id ... |

​```atlas:claims
source_inventory_hash: sha256:<hex>
unit: atlas-wiki -> .codas/wiki
unit: codas-app -> src/codas/app
... (sorted by unit id) ...
roadmap: program:P0:cli-core -> completed
... (sorted by work-item id) ...
​```
```

- **Headings** `## Structure Units`, `## Roadmap` are NOT in the wiki parser's
  `_SECTION_KIND` map → no `wiki_claims`. No `Evidence:` label.
- **Code-span cells** (`` `unit-id` ``, `` `path` ``) render readably; paths are real →
  even if surfaced by `doc_claims`, they exist and `stale_claim` only checks links.
- **`atlas:claims` fence** (` ```atlas:claims `): wiki adapter toggles `in_fence` on any
  ` ``` ` line and skips content → zero claims from it. D3d ships the real parser.

### atlas:claims schema (line-oriented, the D3d contract)

```
source_inventory_hash: <sha256:...>      # exactly one; freshness anchor
unit: <unit-id> -> <path>                # verify structure unit <id> has <path>
roadmap: <work-item-id> -> <status>      # verify program work_item <id> has <status>
```

- One claim per line, `kind: subject -> value`. `->` is a safe delimiter (unit ids,
  work-item ids, and paths never contain it). Lines sorted within each kind; kinds in a
  fixed order (`source_inventory_hash`, then `unit`, then `roadmap`).
- Nonempty by construction (always ≥1 unit). Satisfies design §3 link ③.

## 3. Renderers (pure)

```python
def render_generated_overview(inventory: dict, source_inventory_hash: str) -> str: ...
```

- `units = inventory["units"]` → sorted by `id`; rows + `unit: id -> path` claims.
- `work_items = inventory.get("program", {}).get("work_items", [])` → sorted by `id`;
  rows + `roadmap: id -> status` claims. (Program block may be absent → empty roadmap
  table + no roadmap claims; page still valid.)
- Assemble: header comment + title + preamble + Structure Units table + Roadmap table +
  fenced atlas:claims. Join with `\n`, single trailing newline.
- Pure: reads only the passed dict, returns a str; no I/O, no time.

Markdown table cell escaping: unit ids / paths / statuses are simple tokens (no `|`),
but wrap each in backticks and the renderer asserts no `|`/newline in a cell (defensive,
deterministic). A `|` would be a real bug upstream, not silently mangled.

## 4. Write

```python
GENERATED = ".codas/wiki/generated"

def write_generated_sections(repo: Path) -> list[Path]:
    inv = run_inventory(repo, exclude_under=(GENERATED,))
    h = inventory_hash(render_inventory_json(inv))
    page = repo / GENERATED / "governance.md"
    page.parent.mkdir(parents=True, exist_ok=True)
    page.write_text(render_generated_overview(inv, h))
    return [page]
```

- `write_text` with deterministic content → idempotent (second run rewrites identical
  bytes). The hash is stable across writes because `GENERATED` is excluded from the
  inventory it is computed over (writing the page never feeds its own hash input).
- Returns the written paths so the CLI can report them.

## 5. CLI

`cli.py` `wiki` branch gains `--write` (store_true) before the existing `--emit-pack`
handling:

```python
if args.command == "wiki":
    from .app.wiki import build_atlas_pack, write_generated_sections
    if args.write:
        written = write_generated_sections(repo)
        for path in written:
            print(f"wrote {path.relative_to(repo).as_posix()}")
        return 0
    if args.emit_pack:
        ...
    parser.error("wiki: use --emit-pack or --write ...")
```

`--write` and `--emit-pack` are independent flags (could allow both; keep simple: write
takes precedence, then emit-pack, else usage error).

## 6. Producing + committing the page

After implementing, run `codas wiki --write` to generate
`.codas/wiki/generated/governance.md`, then commit it with the code. The committed page
must be the exact deterministic output (a test regenerates and asserts the on-disk bytes
match `render_generated_overview` — so a hand-edit or staleness is caught even before
D3d).

## 7. Tests (`tests/test_generated_sections.py`)

- `render_generated_overview` pure (input dict unmutated) + deterministic (two calls
  equal) on a hand-crafted minimal inventory; assert the atlas:claims block contains
  `source_inventory_hash:` + the expected `unit:`/`roadmap:` lines, and the headings are
  the neutral ones.
- Idempotency: `write_generated_sections(temp_repo)` twice → file byte-identical. (Use a
  temp repo with a minimal `.codas`; or assert on the real repo that a second write
  equals the committed bytes.)
- Zero wiki_claims: `extract_wiki_claims(repo, (committed page,))` returns no claims
  for the generated page (neutral headings + fenced block).
- On-disk == rendered: the committed `.codas/wiki/generated/governance.md` equals
  `render_generated_overview(excluded-inventory, hash)` (freshness/no-hand-edit guard).
- Real-repo dogfood: `run_check(cwd)` has no findings mentioning the generated page;
  `inventory --json` byte-identical across two builds with the page present.

## 8. Dogfood-safety + determinism recap

- Page under `.codas/wiki` → owned (atlas-wiki unit); neutral headings + fenced claims →
  no wiki_claims; real paths → no stale_claim; no `drift_couplings` target it → no
  spec_drift. `codas check .` = 0 with the page committed (verified empirically).
- `exclude_under=(GENERATED,)` keeps `source_inventory_hash` self-consistent and the
  inventory command byte-identical (default path unchanged).
- New names: `render_generated_overview`, `write_generated_sections`, `GENERATED`
  (already a module constant `_GENERATED_DIR` in wiki.py — reuse it, do not redefine) —
  grep before commit (no `duplicate_implementation`).

## 9. Open questions for review

1. One bundled page (`governance.md`) vs one file per panel. Chosen: one page now
   (simplest, proves the pattern; multi-file is trivial later). Confirm.
2. atlas:claims delimiter `->` vs `=` vs JSON. Chosen line-oriented `kind: subj ->
   value` (human-readable, trivially parseable by D3d, no path-collision). Confirm.
3. Should roadmap claims (`roadmap: id -> status`) be in atlas:claims, given D3d would
   verify them against `program.yml` (a claim source) rather than code facts? They are
   still deterministically checkable. Keep, or restrict atlas:claims to code-fact
   claims (units only) and leave roadmap as display-only? Leaning: keep both (D3d can
   verify program facts too). Confirm.
4. Commit the page in D3b (unguarded until D3d) vs defer committing to D3d. Chosen:
   commit now (design §5 wants it committed for dogfooding; the on-disk==rendered test
   guards hand-edits even pre-D3d). Confirm.
