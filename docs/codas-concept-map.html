§5 · Codas 怎么定位"代码 vs 文档"差异 —— 靠锚点

Codas 读不了散文意思(无 LLM,§17)。它靠"锚点" = 文档里一句机器可核对、指向具体代码事实的引用。

文档里的锚点
defines: 扣款 -> pay.py::Charge::run
→
对照当前事实(层2)
pay.py::Charge::run 还存在吗?
↓ 代码改名 run → execute
锚点解析不上 = 定位到差异
"PRD 第42行锚点指向 Charge::run,代码里没了。" + 配合 diff 猜出 run→execute → 递 RepairTarget 给 AI。
机制干啥读哪严重度状态
code_anchor检测 defines/calls/contains 锚点解析不上.codas/wiki/code/**WARNING(open-world)现状
stale_claim文档路径/符号引用是否还存在markdown 文档WARNING现状
fact_coupling盯的代码变了 → 配套路径必须同提交改,否则拦.codas/claims.yml(手写)ERROR(gate)现状

源头切分(消歧):代码 → 事实(facts),不叫 claim;文档/配置 → claim。代码从不产 claim。"claim" 是一个属(可被事实核对的断言),三个种:① 关系/治理 claim(claims.yml,authored,closed,拦)② doc claim(从文档扫出,核对没说谎)③ 锚点 claim(.codas/wiki/code,open warning)。别裸叫 claim。

关键限制:只有带锚点的散文能被检测;没锚点的纯散文 Codas 看不见。所以锚点要埋。

根治"claim 重载"= 合并不是改名:claims.yml 的 fact_coupling(治理 claim)和文档锚点是同一件事两种写法 → 提案:写一条锚点,既核"文档没说谎"又派生"连改强制"(=问题3)。合了"两种 claim"混淆从根消失。

§6 · open-world vs closed-world(为什么锚点是下界)

代码事实 = open-world

静态读能证"存在",不能证"不存在"(动态/反射/跨语言调用看不见)。
→ 事实是下界:"至少这么多真,可能还有没看见的"。
→ 没观察到 ≠ 不存在。gate 只能基于"确定存在"拦,不能基于"缺失"拦。

配置 = closed-world

structure.yml / claims.yml = 声明,按定义就是全集。
→ 不在 structure.yml 里 = 真的没归属(没说的就是假的)。

这条决定:gate(拦提交)只信确定的 in-core 事实;CodeGraph/多语言事实是 open-world+非确定 → 只配 advisory(提示),不进 gate。
