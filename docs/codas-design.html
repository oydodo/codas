Codas (Code Atlas System) 正式设计方案

Codas 是一个面向长期参与代码维护的 coding agent 的代码库地图与治理系统。 它通过确定性仓库事实、架构索引、任务上下文和强制 gate，阻止重复实现、孤儿文件、文档漂移和未解决的约束冲突进入主分支。

Status Design record (P0–P5 implemented)
Date 2026-06-17 (authored); phase status lives in .codas/program.yml
Audience Agent-maintained repositories
Interface CLI-first, agent-agnostic

1. Executive Summary

Codas is a Code Atlas System for coding agents. It maps a repository into verifiable facts, gives agents task-specific architectural context, blocks changes that introduce duplicate implementations, orphan artifacts, unresolved constraint conflicts, or drift (a change leaving a claim inconsistent with the facts), and surfaces stale claims (inconsistencies no diff can be blamed for) on any run.

Codas 不是 Codex skill、Claude Code 插件、Trellis 插件或某种语言检查器。它是一个 agent-agnostic、language-agnostic、CLI-first 的 repo-local governance layer。Skill、MCP、Claude Code hooks、Codex instructions、GitHub Action 都只是接入方式。

Codas 的核心假设是：agent 不能被当成天然可靠的规则执行者，而应该被当成 untrusted patch producer。因此 Codas 不依赖提示词保证 agent 听话，而是通过 preflight context、local gate、CI gate 和 branch protection 保证违规变更不能静默合入。

2. Problem Statement

在长期使用 coding agent 维护代码库时，常见问题不是单次代码质量，而是结构性漂移：

  • agent 在不知道已有实现的情况下，为同一功能创建第二套类似实现。
  • agent 新增孤儿文件，文件存在但没有被 build、引用、任务或文档解释。
  • agent 修改代码后没有同步 PRD、spec、ADR、repo wiki 或 task 状态。
  • 约束源本身互相冲突，agent 只能猜测哪个文档更可信。
  • 重复实现可能隐藏在私有类型、局部 helper、组件内部结构中，单纯看文件结构无法发现。

这些问题不能靠更长的 prompt 解决。Codas 要把仓库事实变成可查询、可验证、可失败的系统边界。

3. Scope

3.1 适用范围

Codas 适用于 任何让 coding agent 长期参与维护的代码库，并要求 agent 严格遵守其 gate 结果。

Swift / iOS TypeScript / React / Node Python Rust Go Java / Kotlin monorepo backend frontend mobile CLI tools

3.2 非目标

  • Codas 不替代测试、编译、lint、typecheck 或安全扫描。
  • Codas 不替代 Trellis、GitHub Issues、Linear、Jira 等任务系统。
  • Codas 不承诺自动判断所有架构决策是否正确。
  • Codas 不把 LLM 输出当成唯一真相源。
  • Codas 不绑定 Codex、Claude Code、Cursor 或任何单一 agent 产品。

5. Design Principles

  1. Facts over prompts. 可验证事实优先于提示词。
  2. Claims are fallible. PRD、spec、AGENTS、wiki 都可能错，必须能检测冲突。
  3. Agent as untrusted patch producer. agent 可以产出 patch，但 patch 必须通过 gate。
  4. Fail closed for protected branches. 主分支无法确认安全时，默认阻止合并。
  5. Evidence everywhere. 每个 finding 都必须能指向文件、symbol、claim 或 diff。
  6. LLM assists, never rules. LLM 可辅助聚类和解释，但不能作为唯一裁决者。
  7. Adapters at the edge. 语言、构建系统、任务系统差异只能存在于 adapter 层。
  8. Local first, CI enforced. 本地体验要快，最终强约束由 CI 和 branch protection 保证。

9.4 Fact Soundness & the Open-World Invariant

每条 emitted fact 都是 SOUND 的（提取器保守，只吐它解析出的，丢弃不确定的）， 所以对 fact 的正面读取 100% 可信（"A 调用 B" 为真）。但静态代码分析得到的家族 （symbols / imports / calls，以及正则抽取的 doc/wiki 路径引用）是 OPEN-WORLD：它们是一个 sound 的下界，而非完整集。缺席不是证据—— "没有 A→B 这条边" 不代表 A 从不调用 B，可能是静态分析无法判定的动态/反射形式（完整性对运行时行为不可判定，Rice 定理）。 相对地，config/声明类家族（structure units、tasks、work-items、documents）是 CLOSED-WORLD： 整份读入，所以它们的缺席即证据（"structure.yml 未声明 unit X" ⇒ X 确实未声明）。

不变量的两个后果：

  1. Policy MUST NOT 在 open-world fact 的缺席上设 gate —— 确定性 policy 只在 存在上判（duplicate_implementation、dependency_direction）。刻意没有 "dead code" gate， 因为它要判 calls 边的缺席，会在动态派发上误报。
  2. 在这些 fact 上推理的消费者（今天的 codas impact；之后判定语义合法性的 LLM / claim verifier） MUST 把缺席读作未知，而非否定。

曾被推迟的 per-family open/closed 标记现在有了消费者（W3 calibrator，src/codas/app/calibrate.py）， 故以静态注册表落地：src/codas/facts/openworld.py 的 WORLD_BY_FAMILY （symbols/imports/calls/contains = open；units/tasks/work_items/documents = closed） 与 world_of(family) 访问器（open|closed|None）。calibrator 据此给结构 claim 评级：open 家族缺匹配 ⇒ UNCONFIRMED（绝不读作"假"），closed 家族缺匹配 ⇒ CONTRADICTED，未知家族 ⇒ SEMANTIC。 这是静态常量，绝不序列化进 byte-identical inventory（不从空的 open_world_gaps() 推断 closed-world —— 空元组不是 closed-world 断言）。

完整的感知模型（fact 家族、结构化 claim schema、把语义合法性判定交给 LLM 的 engine-split、 Truth-Maintenance-System 血脉、双向传播引擎）是活文档，记录在 CONTEXT.md 的 Perception Model 节 与 06-20 perception-model 决策记录中；本节只钉死其中稳定的 open-world 不变量，细节以那两处为准（anchor-to-source，避免复制后再漂）。 早先的分级 "soundness qualifier"（exact/scoped/approximate）已退役 —— 静态代码 fact 一律 open-world， 分级无信息；操作含义只有 open/closed 二元。

11. Policy and Gates

11.1 基础 Policy

权威的实现——即实际接线运行的检查集合——是 src/codas/app/check.py。 .codas/policies.yml 声明治理 policy 及其 severity，但并不等于“已接线集合”：它没有声明 check.py 同样接线的 bootstrap / meta 与 load 检查（config_sources、dogfooding、trellis_context、structure_map、 program_plan、document_set、waivers），而且它仍声明了一些尚未实现（declared-but-planned）的 policy （duplicate_concept、orphan_artifact、missing_canonical_owner、constraint_conflict、stale_preflight）。 因此本节不再枚举完整清单以避免再次漂移；当前实际接线并运行的治理与 meta 检查包括： config_sources、dogfooding、trellis_context、structure_map、missing_structure_owner、 structure_drift、deprecated_path_used、program_plan、document_set、stale_claim、 duplicate_symbol、duplicate_implementation、dependency_direction、stale_wiki_claim、 fact_coupling、generated_wiki_drift、waivers、policy_registry。

headline policy（severity 请以 .codas/policies.yml 为准；接线状态以 check.py 为准）：

Policy 目标 默认级别
duplicate_implementation 同一 top-level symbol 在多个 src 模块实现时，必须在 .codas/claims.yml 声明 canonical/variant/migration 关系，否则按未声明重复实现报错。 Error
structure_drift 仓库变更必须与 Structure Map 对齐，或显式更新它。 Error
fact_coupling fact 级 co-change 耦合（.codas/claims.yml 的 fact_couplings）：当被监视的 symbol/import/call add/remove（working-tree-vs-HEAD fact-delta）出现时，required 伴随路径必须在同一 diff 中 co-change。fact 触发 → always-true（注释修改不产生 fact-delta）。取代 v1 spec_drift/drift_couplings。 Error
stale_claim 发现文档/spec 中不存在或已移动的路径和 symbol。 Warning
stale_wiki_claim Atlas Wiki claim 必须对照仓库 facts 验证（canonical-source / evidence / sync_target 路径存在性）。 Warning
generated_wiki_drift committed generated Atlas page 的 atlas:claims 必须与 Structure Map / Program Plan facts 一致；被 facts 反驳的 claim 报错。（名字带 drift,但实为 STATE/STALE 检查:claim-vs-facts,无需 diff;非 fact_coupling 那种 diff 触发的 DRIFT。） Error
fact_coupling = materiality-as-coupling-property（决策记录，spec-drift v2）： materiality 是 耦合 的静态属性（作者写一次），不是逐次变更的判断（§5「LLM assists, never rules」/ §15 LLM Usage 禁止后者进核心）。 粗粒度的 must_update_if_changed（“unit 变 → doc 变”）没有 always-true 的 fact 级形式，保持 advisory； 真正的 gate 是 .codas/claims.yml 的 fact_couplings，触发器是 fact-delta （symbol/import/call 的 add/remove，working-tree-vs-HEAD），故 always-true by construction：注释修改不产生 fact-delta，耦合 dormant。 这是 Drift/Staleness 2×2 的 DRIFT 半（changed+inconsistent，commit 时 diff 可归因）；互补的 STALE 半（unchanged+inconsistent，无 diff 可归因）由状态政策 policy_registry / generated_wiki_drift / stale_claim / stale_wiki_claim 抓。两半同病、两个时刻,缺一不可。canonical 定义见 CONTEXT.md。 v1 spec_drift（file 级 drift_couplings，vouched）已退役；substrate 见 06-20-fact-delta-substrate，gate 见 06-20-fact-coupling-gate。

11.2 Gate Levels

Gate 命令 执行时机 目的
Preflight codas preflight . --task <task> agent 开工前 给出已有实现、相关文档、canonical owner、重复风险和必须同步项。
Local Check codas check . 提交前或工作完成后 尽早发现 duplicate、orphan、drift、conflict。
CI Check codas check . --ci --base origin/main PR / merge 前 作为 required status check 阻止违规变更进入 protected branch。
Review Report codas report . --format markdown PR review 解释 finding 和推荐修复路径。
强制原则：如果仓库启用了 Codas strict mode，CI gate 中的 Error finding MUST 阻止合并。agent 不能通过解释或忽略绕过 gate，只能修复、更新约束源或提交有效 waiver。

12. Compliance Model

Codas 的遵守机制分四层，越往后越硬。

层级 机制 保证
Context agent 开工前读取 context pack。 降低误写概率，但不构成强保证。
Wrapper codas run --task <task> -- <agent-command> 自动 preflight 和 check，记录 receipt。
Local Hook pre-commit / pre-push 运行 codas check。 阻止大多数本地坏提交，可被手动绕过。
Remote Gate CI required status check + branch protection。 真正强保证。绕过本地 hook 的变更也不能合并。

因此 Codas 的安全边界不是“agent 一定会听话”，而是“即使 agent 不听话，违规 patch 也无法静默进入主分支”。

13. Constraint Conflict Resolution

当 constraint source 互相冲突时，Codas 不让 agent 猜测，而是产生 error finding。

13.1 参考优先级

  1. Build graph 和 package graph：实际进入构建、测试、发布的事实。
  2. Current code facts：当前代码里真实存在的 symbol、reference、module。
  3. Explicit Codas overrides：仓库内显式声明的 canonical owner 或 policy override。
  4. Architecture docs / ADR：架构决策。
  5. Spec / PRD / task docs：功能与任务约束。
  6. Agent instructions：AGENTS、CLAUDE、Codex skill 等行为约束。
  7. Generated wiki：辅助导航，不能作为最高权威。

优先级只用于诊断和建议，不用于自动吞掉冲突。只要两个约束源都声称自己是 authoritative， Codas MUST 要求人类或明确 waiver 做决策。 

CONSTRAINT_CONFLICT
concept: ComposerSuggestionPopup
claims:
  - source: docs/specs/composer.md
    claim: canonical owner is ChatInputComposer
  - source: code facts
    claim: another implementation exists in GlobalComposerBar
required resolution:
  1. extract a shared component
  2. mark one implementation deprecated
  3. update the canonical owner claim
  4. or create a temporary waiver with an expiry date

15. LLM Usage

适合 LLM 不应只靠 LLM
给 concept 起名和聚类。 判断路径是否存在。
总结 wiki 页面。 判断文件是否进入 build。
解释 finding 和推荐修复方案。 判断 symbol 是否定义。
推断两个实现是否语义相近，并给出置信度。 判断 git diff 触碰了哪些 artifact。

LLM 输出必须降级为 claim 或 suggestion，不能直接成为 fact。所有 LLM-assisted finding SHOULD 携带置信度和 evidence。

语义合法性（"这是对的抽象吗"、"代码是否真的实现了它声称的目的"、"两个概念是否重复"的语义判断）是 LLM 的职责， 基于 Codas 的确定性 fact 进行（ground truth 钉 fact，不钉 LLM 自己生成的 wiki —— 否则是 LLM 查 LLM，共享幻觉）。 关键护栏 = §9.4 的 open-world 不变量：LLM 在代码 fact 上推理时 MUST NOT 把"fact 没说"读作"事实否定" （否则会把"无调用者"误判成死代码 —— 缺席只是下界）。

17. Waiver Model

Waiver 是显式豁免，不是忽略规则。每个 waiver 必须可审查、可过期、可归责。

waivers:
  - id: duplicate-concept-composer-popup-2026-06
    finding: duplicate_concept
    concept: ComposerSuggestionPopup
    reason: Temporary migration; old implementation will be removed.
    owner: "@oydodo"
    expires: "2026-07-15"
    evidence:
      - path: src/ui/composer/Composer.tsx
      - path: src/ui/global/GlobalComposerBar.tsx
  • 过期 waiver MUST 在 CI 中失败。
  • 没有 owner 的 waiver MUST 无效。
  • 没有 reason 的 waiver MUST 无效。
  • strict mode 下 error finding MUST 只能通过修复或有效 waiver 解除。

18. Agent Integrations

Codex

通过 Codas skill 和 CLI 接入。skill 负责教 Codex 在开工前运行 preflight、完成后运行 check，并读取 context pack。

Claude Code

通过 CLAUDE.md、slash command 和 hooks 接入。hooks 可在任务开始和提交前调用 Codas。

Cursor / Other Agents

通过 CLI、MCP server 或 repo-local wiki 接入。只要能执行 shell 或调用 MCP，就能使用 Codas。

CI / Review

通过 GitHub Action、GitLab CI、Buildkite 或任意 CI 运行 codas check --ci，作为 required status check。

19. Workflow

19.1 Agent 开工流程

  1. 用户或任务系统给出 task。
  2. agent 或 wrapper 执行 codas preflight . --task <task>。
  3. Codas 生成 context pack，列出 canonical owner、相关实现、相关文档、重复风险。
  4. agent 基于 context pack 修改代码。
  5. agent 执行 codas check .。
  6. 如果有 error finding，agent 必须修复、更新约束源或添加有效 waiver。
  7. PR 中 CI 执行 codas check . --ci，失败则禁止合并。

19.3 Project Planning Documents

Trellis owns task-level workflow, but Codas also needs a project-level planning layer. A governed repository MUST distinguish the role of each planning document instead of relying on filename convention or chat memory.

Document Role Default Carrier Responsibility
Product / Architecture Design docs/codas-design.html Defines product scope, architecture principles, domain terms and non-goals.
Implementation Plan docs/codas-implementation-plan.html Translates design into module boundaries, phases, commands, policies and test strategy.
Structure Map Schema docs/codas-structure-map-schema.html Defines Structure Map fields, validation model, role workflows and normalized structure inventory shape.
Program Plan .codas/program.yml or a linked roadmap document Defines milestones, dependencies, sequencing, work items and phase exit criteria.
Document Role Manifest .codas/documents.yml Defines the Project Document Set: required docs, paths, owners, authority and update triggers.
Structure Map .codas/structure.yml Defines Repository Structure, ownership, canonical placement and structural update obligations.
Task PRD / Context .trellis/tasks/** Defines requirements, implementation context and verification context for a single work item.
Spec / Guidelines .trellis/spec/** Defines recurring engineering rules and quality checks learned across tasks.
Orientation .codas/wiki/** Helps agents navigate concepts and decisions, but does not become a fact source.

The Document Steward owns the Document Role Manifest. During project bootstrap this role may be performed by the Structure Architect, because document placement and authority are part of the initial Repository Structure. During ongoing work, Codas should check that substantial Trellis tasks link to an approved Program Plan phase or work item.

20. Dogfooding Protocol

Codas must govern Codas. The Codas repository is the first governed workspace. During implementation, every agent working on Codas must follow the same philosophy Codas is designed to enforce for other repositories.

Before Codas can fully run its own preflight and check commands, the repository uses a bootstrap protocol. This protocol is intentionally manual but mirrors the future system boundary: identify the concept being changed, read canonical sources first, update related claims, run available gates, and leave evidence for the next agent.

20.1 Bootstrap Rules

  1. Every implementation task MUST name the affected concept before editing.
  2. Substantial implementation work MUST identify the Program Plan phase or work item it advances once a Program Plan exists.
  3. Every task MUST read docs/codas-design.html and the relevant .codas/wiki/** page before changing code.
  4. Changes to behavior, architecture, command shape, policies or data model MUST update the design document or wiki claim that describes them.
  5. New files MUST have an explained owner: core, adapter, policy, integration, docs or tests.
  6. Temporary duplication MUST be represented as a waiver with owner, reason and expiry.
  7. At the end of each task, the agent MUST run the available local gate and report the result.

20.2 Bootstrap Gate

PYTHONPATH=src python3 -m unittest discover -s tests
git status --short

Until codas check exists, this bootstrap gate is not a full substitute for Codas. It is a minimum receipt: tests verify current behavior, and git status reveals newly created artifacts that must be explained in the final report.

20.3 Dogfooding Acceptance

  • Codas implementation work MUST NOT introduce architecture claims only in chat.
  • Codas implementation work MUST NOT leave a new source, test or docs artifact without an owner and purpose.
  • Codas implementation work SHOULD convert repeated manual dogfooding steps into product features.
  • The first mature version of codas check MUST be able to check this repository.

20.4 Trellis Task System

The Codas repository uses Trellis as its canonical task system. Trellis owns task lifecycle, persisted requirements and task context; Codas consumes Trellis as a workflow adapter and task fact source.

System Owns Does Not Own
Trellis Task creation, prd.md, implement.jsonl, check.jsonl, workflow phases. Architecture inventory, policy gates, waivers, duplicate detection.
Codas Atlas inventory, facts, claims, concept index, policy gates, waivers, Atlas Wiki. Replacing Trellis as the project task workflow.

Substantial implementation work in this repository MUST have a Trellis task under .trellis/tasks/. Requirement changes MUST be persisted in that task's prd.md, not left only in chat.

22. Acceptance Criteria

  • codas inventory 能在无语言 adapter 的情况下生成基础 artifact 和 doc claim 索引。
  • codas check 能拦 DRIFT（changed+inconsistent:fact_coupling 的 co-change gate、structure_drift 的边界）并查出 STALE（unchanged+inconsistent:stale_claim / stale_wiki_claim / generated_wiki_drift / policy_registry），外加未声明的 duplicate implementation 等已接线违规（orphan_artifact 为 declared-but-planned，尚未接线；v1 spec_drift 已被 fact_coupling 取代）。
  • codas preflight 能基于 task 输出 read-first 文件、相关 concepts 和 duplicate risks。
  • codas wiki 生成的页面每条 canonical 结论都包含 evidence。
  • strict mode 下 error finding 会导致非零退出码。
  • waiver 必须带 owner、reason、expires，否则无效。
  • CI 示例能作为 required status check 使用。
  • 核心模型不包含 Swift/iOS 专有概念，Swift 只作为 adapter 存在。

23. Initial Migration from Current Prototype

旧 harness-guard 原型不再作为兼容入口保留。迁移到 Codas 的方式如下：

  1. 删除旧 package 和 CLI：harness_guard、harness-guard。
  2. 建立新的 package 和 CLI：codas。
  3. 把已有 checks 拆成通用 policy：ignored/buildable、duplicate type、stale doc path、code/spec 一致性（v2 由 fact_coupling 的 co-change DRIFT gate 实现;v1 spec_drift 已退役）。
  4. 把 Swift/XcodeGen 逻辑移动到 adapter 层。
  5. 新增 .codas/config.yml、policies.yml、waivers.yml。
  6. 新增 inventory、preflight、wiki 子命令。
  7. 保留 Ciri 作为第一个 validation repo，但不让 Ciri 的结构进入核心抽象。

24. Risks and Mitigations

风险 缓解
误报过多导致团队关闭 Codas。 默认 warning 起步，strict mode 分阶段启用，支持可过期 waiver。
LLM 聚类不稳定。 LLM 结果只作为 suggestion，必须带 evidence 和 confidence。
多语言 monorepo adapter 复杂。 核心先做 artifact/doc/git 通用能力，语言能力按 adapter 增量加入。
agent 绕过本地流程。 CI required status check 和 branch protection 作为最终强约束。
wiki 变成另一份漂移文档。 wiki 从 inventory 生成，每条关键结论要求 evidence 和 last verified。

25. Normative Requirements

  • Codas MUST be agent-agnostic.
  • Codas MUST be language-agnostic at the core layer.
  • Adapters MUST convert ecosystem-specific facts into the shared data model.
  • Findings MUST include evidence.
  • Strict CI mode MUST fail on unresolved error findings.
  • Waivers MUST include owner, reason and expiry.
  • Atlas Wiki MUST NOT be treated as the highest authority.
  • LLM output MUST NOT be stored as fact unless independently verified.
  • Codas implementation work MUST follow the Dogfooding Protocol until Codas can enforce itself.
  • Codas implementation work MUST use Trellis as the repository task system.
  • Agent integrations SHOULD run preflight before code edits.
  • Protected branches SHOULD require Codas CI status checks.
