Codas 实施方案：架构与模块设计

1. Executive Decision

Implementation should start with the fact and policy core, not with hooks or wiki. Hooks, wiki, MCP and CI are delivery channels. The product becomes useful only after Codas can convert a repo into verifiable facts, reconcile claims, run policies and emit evidence-backed findings.

The first implementation milestone is P0: Codas CLI Core and Self-Check. It creates a real codas entrypoint, reads .codas/config.yml, scans declared authoritative sources, validates Trellis task facts and removes the legacy harness_guard prototype.

2. Architecture Principle

The central product chain is:

Repository facts
  -> authored claims
  -> verified governance facts
  -> policies
  -> findings
  -> receipts
  -> agent / CI enforcement

This chain keeps the implementation honest. The repository is the fact source. Wiki, PRD, specs and Trellis tasks are claim and workflow sources. Codas reconciles claims against facts before using them for governance.

Important: Wiki is not the truth source. Wiki is a readable summary and claim surface. A wiki claim becomes a governance fact only when Codas can verify its evidence and authority.

Dependencies point downward. Interfaces may call application services. Application services orchestrate engines. Engines depend on core domain models and adapter protocols. Core domain must not import Swift, Trellis, GitHub, Codex, Claude Code or any product-specific integration. The ScanContext fact-provider seam (codas.facts.context) is the single normalization layer between adapters and core/policies: it runs the adapters and exposes their output as facts and claims, so that core and policies never import codas.adapters. The dependency_direction policy dogfood-enforces this boundary.

Source Inventory Hash

{
  "source_inventory_hash": "sha256:...",
  "computed_over": "inventory EXCLUDING .codas/wiki/generated/",
  "embedded_in": ".codas/wiki/generated/*.md (atlas:claims block)",
  "verified_by": ["generated_wiki_drift policy", "codas wiki --verify"]
}

A source_inventory_hash pins a generated wiki page to the inventory it was rendered from. It is computed over the inventory excluding .codas/wiki/generated/ to avoid a self-reference loop (the inventory ingests .codas/wiki/). The committed page narrows the hash further to only the fields it renders, so unrelated edits do not restale it. The generated_wiki_drift policy checks that each committed page carries a nonempty atlas:claims block with this hash; codas wiki --verify verifies its freshness.

These contracts should start as Python dataclasses. JSON schema can follow once the shape stabilizes.

7. Command Design

Command Purpose P0 Behavior
codas check . Run configured policies and return non-zero on errors. Self-check config, Trellis task facts, stale paths and waivers.
codas inventory . Build Atlas inventory. Emit basic artifact and claim index JSON.
codas preflight . --task <task> Generate task-specific context for agents. Stub or read-only task summary until inventory matures.
codas wiki . Generate or verify Atlas Wiki. Verify existing wiki paths; generation later.
codas structure . Validate Structure Map and report structural drift. Later.
codas plan . Validate Program Plan, Document Role Manifest and task alignment. Later.
codas explain . --finding <id> Explain evidence and repair choices. Later.
codas doctor [--json] Diagnose Codas installation and integration gaps. Implemented (P6 S1). Read-only diagnostics: config/policies/waivers/structure load, config-declared optional sources (program/documents), Trellis tasks root, git work tree, removed-prototype leftovers. Exits 1 on any fail. codas init (P6 S3) scaffolds the inverse minimal .codas/ skeleton (config/policies/structure/waivers), no-clobber by default.

9. P0 Detailed Slice

P0 should be small enough to complete without rewriting the whole prototype.

  1. Create src/codas/ with CLI, app services and core dataclasses.
  2. Add scripts/codas and [project.scripts] codas = "codas.__main__:main".
  3. Remove the legacy harness_guard package and scripts/harness-guard.
  4. Implement config loader for .codas/config.yml, policies.yml and waivers.yml.
  5. Implement filesystem checks for declared authoritative paths.
  6. Implement Trellis task context scan for task.json, prd.md, implement.jsonl and check.jsonl.
  7. Implement finding output with evidence and JSON mode.
  8. Add tests for Codas self-check fixtures.

P0 Non-Goals

  • No full wiki generation.
  • No MCP server.
  • No CI action.
  • No LLM-assisted concept clustering.
  • No complete Swift adapter rewrite unless needed to avoid core pollution.

10. Policy Roadmap

Policy First Implementation Later Expansion
declared_source_exists All configured authoritative/supporting source globs resolve as intended. Ownership and freshness checks.
stale_claim Markdown path references point to existing files. Symbol and fragment anchors.
trellis_context_complete Tasks include expected PRD and context files. Current task freshness and receipt matching.
document_set_complete Required Document Roles resolve to existing files with authority, owner and update triggers. Staleness checks and missing synchronization reminders.

waiver_valid Waivers have id, owner, reason and expiry. Waiver-to-finding matching and expiry enforcement.
orphan_artifact New files under changed diff have config/spec/task explanation. Language-aware reference graph.
missing_owner Artifacts in governed paths match a Structure Unit with ownership. Owner freshness, escalation routing and role assignment.
structure_drift Changed paths remain inside declared Repository Structure boundaries. Dependency direction, deprecated paths and migration windows.
duplicate_implementation Repeated symbols or concepts require canonical, variant or migration claims. Language-aware and LLM-assisted concept similarity.
duplicate_symbol Language adapters emit repeated type/function names. Concept-level semantic duplicate detection.
constraint_conflict Same subject has incompatible authoritative claims. Conflict workflow and repair suggestions.

11. Adapter Boundary

Adapters translate ecosystem-specific reality into shared Codas contracts.

Adapter Allowed To Know Must Emit
Markdown Markdown links, code spans, frontmatter, headings. path claims, anchor claims, authored decisions.
Trellis .trellis workflow, tasks, archives, JSONL context. workflow facts, task facts, requirement claims.
Python modules, classes, functions, imports, pyproject. symbol facts, package facts.
Wiki Markdown wiki pages, atlas:claims fences, authored unit/roadmap claims. wiki_claims facts (authority + evidence for reconciliation).
Callgraph first-party Python call edges via stdlib ast (resolution-tagged; pyan rejected for cross-process nondeterminism). deterministic call facts (the inventory dependency graph; substrate for impact and spec_drift).
Git working-tree diff and changed paths, without mutating the repo. changed_paths substrate (diff grounding for spec_drift).
Planned per-language adapters (future — illustrate the boundary; not built today)
Swift (future) Swift files, declarations, extensions, imports. symbol facts, reference facts.
XcodeGen (future) project.yml targets and source roots. build inclusion facts.
TypeScript (future) tsconfig, package.json, exports, components. symbol facts, dependency facts.

The rows above the divider are the adapters that ship today (markdown, trellis, python, wiki, callgraph, git — only Python carries per-language symbol extraction). The Swift/XcodeGen/TypeScript rows are illustrative: the §11 boundary is multi-language by design, but those per-language adapters are not yet built.

Boundary rule: if a concept only makes sense in one ecosystem, it belongs in an adapter. Core may only receive normalized facts and claims.

Generated inventory and reports may be either committed or ignored per repo policy. 14. Engineering Order of Operations

  1. Open a Trellis task for P0 migration.
  2. Add Codas package shell and tests for command dispatch.
  3. Implement config loader and core models.
  4. Implement file/artifact scanner for declared sources.
  5. Implement Trellis adapter for active and archived tasks.
  6. Define Program Plan and Document Role Manifest schemas before enforcing project-level planning gates.
  7. Implement first policies and JSON/console reports.
  8. Wire codas check . to run the P0 policies.
  9. Run Codas against this repo and fix any dogfooding findings.

15. Open Product Decisions

Decision Default Recommendation Why
Should inventory JSON be committed? Not by default for P0. Keep generated churn low until schema stabilizes.
Should hooks be installed automatically? No. Generate hook snippets first; CI can enforce later.
Should wiki pages be generated or authored? Mixed, but claims must be structured and verified. Pure generated wiki lacks intent; pure authored wiki drifts.
Should LLM clustering be in MVP? No. Need deterministic fact foundation first.
Should Trellis script behavior be patched? Only if it blocks Codas tasks. Trellis is a dependency/workflow adapter; avoid owning it too early.
Should Program Plan and Document Role Manifest be YAML-only? No final decision in this task. This task defines document governance semantics; concrete schemas should be finalized with validation fixtures.

16. Acceptance Criteria for P0

  • codas --help shows Codas commands.
  • codas check . reads .codas/config.yml.
  • codas check . --json emits findings with evidence.
  • codas check . validates authoritative source paths.
  • codas check . reads Trellis active and archived task context files.
  • Swift/Ciri-specific constants are not introduced into codas.core.
  • Unit tests and bootstrap gate pass.

17. What Not To Build Yet

  • Do not start with hook installation. A hook without a reliable policy engine only creates noise.
  • Do not start with full wiki generation. Wiki must follow inventory, not precede it.
  • Do not start with MCP. Agent query UX is valuable only after the underlying facts are trustworthy.
  • Do not overfit Ciri. Ciri is a validation repo, not the Codas domain model.
  • Do not depend on LLM similarity for P0 correctness.
This implementation plan is an authoritative Codas source. Update it when architecture, modules, phase ordering or acceptance criteria change.
