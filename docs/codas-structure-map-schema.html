Codas Structure Map Schema

Structure Map 是 Repository Structure 的可验证声明载体。它让 agent 在新增文件、移动模块、 抽象组件或实现新能力前，先知道 canonical placement、owner、边界和必须同步的 claim source。

Status Schema Design
Date 2026-06-17
Authored Form .codas/structure.yml
Normalized Form .codas/inventory/structure.json

1. Contract

Structure Map is the governed structure claim surface. It is authored in the repo, reviewed like code, and later reconciled against observed repository facts.

Structure Map answers questions that PRD/spec files usually do not answer: where a capability belongs, which module owns it, which paths are canonical, which paths are deprecated, and which documents must change when a structure unit changes.

Structure Map is not the repository fact source. The fact source is still the repository itself. Codas turns Structure Map entries into Claims, verifies them against observed facts, and accepts only reconciled claims as Governance Facts.

2. Authority Model

Repository files
  -> Observed Facts

.codas/structure.yml
  -> Structure Claims

Observed Facts + Structure Claims
  -> Governance Facts
  -> structure policies
  -> Findings
  -> agent / CI gates
  • .codas/structure.yml MUST be treated as an authoritative claim source.
  • .codas/inventory/structure.json MUST be generated or normalized, not hand-authored.
  • Atlas Wiki MUST NOT override Structure Map. Wiki can explain and link to it.
  • Waivers MUST represent temporary exceptions to structure findings, not silent drift.

3. Authored YAML Shape

The first schema uses a keyed map instead of a list of anonymous entries. Stable IDs make findings, waivers and future migrations easier to target. 

version: 1
kind: structure_map
metadata:
  project: Codas
  schema: docs/codas-structure-map-schema.html
  updated: 2026-06-17

defaults:
  status: active
  required_evidence: [path]

units:
  codas-core:
    path: src/codas
    kind: package
    owner: Codas Core
    purpose: Agent-agnostic CLI and governance core.
    canonical_placement: Codas runtime code lives under src/codas.
    status: active
    allowed_children:
      - codas-app
      - codas-core-models
      - codas-config
      - codas-policies
      - codas-reporting
    must_update_if_changed:
      - docs/codas-implementation-plan.html
      - .codas/wiki/concepts/codas-product.md

dependency_rules:
  codas-core:
    may_depend_on:
      - codas-docs
    must_not_depend_on:
      - agent-integrations

deprecated_paths:
  harness-guard:
    path: src/harness_guard
    replacement: src/codas
    status: removed

4. Structure Unit Fields

Field Required Meaning
path Yes Repo-relative path or glob owned by this unit.
kind Yes Directory, package, docs area, task area, policy module, adapter, integration or other repo-specific unit kind.
owner Yes Responsible module, team, domain role or maintained boundary.
purpose Yes Short explanation of what belongs here and why this unit exists.
canonical_placement Yes Placement rule that helps agents decide whether new work belongs here.
status No active, planned, deprecated, removed or external.
allowed_children No Child structure units expected under this boundary.
may_depend_on No Allowed unit dependencies when dependency rules are local to the unit.
must_not_depend_on No Forbidden unit dependencies.
must_update_if_changed No Docs, specs, wiki pages or config files that should change when this unit changes materially.
evidence No Optional paths or claim anchors that justify the structure claim.

5. Normalized JSON Shape

Future codas inventory should normalize authored entries into deterministic JSON. The normalized form keeps exact evidence, resolved paths and derived relationships for policy checks. 

{
  "schema_version": 1,
  "source": ".codas/structure.yml",
  "units": [
    {
      "id": "codas-core",
      "path": "src/codas",
      "kind": "package",
      "owner": "Codas Core",
      "status": "active",
      "claims": [
        "claim:structure:codas-core:canonical_placement"
      ],
      "observed": {
        "exists": true,
        "artifact_count": 14
      },
      "must_update_if_changed": [
        "docs/codas-implementation-plan.html",
        ".codas/wiki/concepts/codas-product.md"
      ]
    }
  ],
  "conflicts": []
}

6. Structure Architect Workflow

  1. Inventory the current top-level repository areas and existing implementation modules.
  2. Draft Structure Units with path, kind, owner, purpose and canonical placement.
  3. Mark deprecated or removed paths explicitly instead of deleting their history from the map.
  4. Define required update obligations for docs, specs, wiki and task-system files.
  5. Run available Codas checks and resolve missing source or stale-claim findings.

The Architect optimizes for future agent navigation. A good map is not exhaustive on day one, but every active top-level area should have an owner and placement story.

7. Structure Steward Workflow

  1. Before editing, identify the target Structure Unit for the change.
  2. If no unit owns the target path, add a unit, move the work, or create a waiver before implementation.
  3. When adding a new capability, check canonical placement and duplicate implementation risks.
  4. When moving or deleting files, update deprecated paths, required update obligations and related wiki navigation.
  5. After implementation, run codas check and codas inventory.

The Steward is allowed to change the Structure Map, but not silently. A structure change is a claim change and must be reviewed with evidence.

8. Policy Checks

Policy Initial Rule Finding
structure_map_loads .codas/structure.yml parses and has version, kind and units. Error when the map cannot be read.
missing_structure_owner Changed artifacts under governed paths match a unit with owner. Error with path evidence and nearest candidate units.
structure_drift Changed paths remain inside active structure boundaries and update required claims. Error for unmanaged drift, warning for incomplete synchronization.
duplicate_implementation New symbols or concepts require canonical, variant or migration relationship. Error when a second implementation lacks a declared relationship.
deprecated_path_used New files must not be added under deprecated or removed paths. Error with replacement path when known.

9. Non-Goals

  • Structure Map does not replace tests, build config, package manifests or language parsers.
  • Structure Map does not decide architecture quality by itself. It makes structure claims explicit and checkable.
  • Structure Map does not bind Codas to Codex, Claude Code or any specific coding agent.
  • Atlas Wiki does not become the authority for structure. It links to the map and explains it.

10. First Implementation Slice

  1. Create .codas/structure.yml for this repository.
  2. Add docs/codas-structure-map-schema.html to authoritative sources.
  3. Add .codas/structure.yml to authoritative sources.
  4. Add Atlas Wiki navigation for Repository Structure.
  5. Load the map via codas inventory . (normalized Atlas JSON) and the structure_map_loads check; later phases add structure-drift findings.
